כך נמנעים מהצטברות שינויים גדולים מדי, קונפליקטים מיותרים, וסטייה מהאפיון.

## איך לעבוד מול Codex בפועל

### כלל עבודה מרכזי
לא לתת ל-Codex את כל הקבצים וכל המשימות בבת אחת.

במקום זה:
- לעבוד מקובץ bundle אחד בלבד
- ובתוך כל bundle אפשר גם לעבוד TASK אחרי TASK
- או לתת את כל המקבץ אם הוא עדיין קטן וברור

### המלצה מעשית
הדרך הבטוחה ביותר היא:

1. לפתוח את קובץ המקבץ
2. לקחת את `TASK 1`
3. להזין אותו ל-Codex
4. לבדוק את הפלט
5. לאשר / לתקן
6. לעבור ל-`TASK 2`
7. להמשיך עד סוף המקבץ
8. רק אז לבצע commit/push/pull

---

## Review מומלץ אחרי כל TASK
אחרי כל משימה ש-Codex מבצע, מומלץ לבדוק:

- האם השינוי תואם במדויק למשימה
- האם נוספו קבצים מיותרים
- האם יש חריגה מהאפיון
- האם יש over-engineering
- האם שמות הקבצים והפונקציות עקביים
- האם אין קוד כפול
- האם אין תלות מיותרת

---

## Review מומלץ אחרי כל Bundle
אחרי שסיימת מקבץ שלם, מומלץ לבדוק:

- האם המקבץ יצר תוצאה קוהרנטית
- האם אין שבירה של מה שהיה קודם
- האם ה-README עדיין נכון
- האם אפשר להריץ את הפרויקט
- האם אפשר להבין את הזרימה מקצה לקצה

---

## תהליך Git מומלץ אחרי כל Bundle
סדר הפעולות המומלץ:

1. `git status`
2. לעבור על הקבצים שהשתנו
3. להריץ בדיקות רלוונטיות אם יש
4. `git add ...`
5. `git commit -m "..."` עם הודעה ברורה
6. `git push`
7. אצלך מקומית לבצע pull / sync
8. לבדוק שהכל התעדכן תקין

## דוגמאות להודעות commit
אפשר להשתמש בהודעות כגון:

- `chore: normalize project structure and repo setup`
- `feat: implement iris prediction model and service flow`
- `feat: add prediction form and result rendering`
- `test: add unittest coverage for validation service and routes`
- `feat: add optional demo enhancements`

## מתי לעצור ולא להמשיך למקבץ הבא
לא להמשיך ל-bundle הבא אם אחד מהדברים הבאים קורה:

- יש שגיאות ריצה
- יש בדיקות נכשלות
- Codex הוסיף דברים שלא ביקשת
- שכבה חדשה נבנתה על בסיס קוד לא יציב
- התיעוד כבר לא תואם למצב הריפו

במצב כזה:
- מתקנים קודם
- מבצעים commit נקי
- ורק אז ממשיכים
